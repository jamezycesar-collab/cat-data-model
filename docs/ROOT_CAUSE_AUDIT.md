# Root-Cause Audit: How 24 Fabricated CAT Event Codes Reached Production

## Summary

The v2.0.0 push of `cat-pretrade-data-model` shipped a reference table that claimed to enumerate "all 50 CAT events" per "FINRA CAT Industry Member Technical Specifications v4.1.0r9." Verification against the actual spec (v4.1.0r15, the closest version available at audit time) showed:

- The current spec defines **99** event types (39 equity + 35 simple option + 25 multi-leg). The figure 50 has no basis in any version of the spec.
- The shipped reference table contained **49** rows, of which:
  - **25 codes are real** and appear in the spec
  - **24 codes are fabricated** and appear nowhere in any version of the spec
- The shipped model has **0 of 25** multi-leg event codes — the entire family is absent.
- Effective coverage of the real spec: **25.3%**.

This document traces the failure modes, identifies the systemic conditions that allowed them, and pins the controls that prevent recurrence.

## The five failure modes

### 1. No primary source on disk during construction

The reference table was authored without the spec PDF in the working directory. Construction relied on:

- AI training-data recollection of CAT event codes
- Search-result snippets from secondary sources
- Pattern extension from a small set of seed codes

None of these are authoritative. There was no point at which a contributor was expected to open Table 15 of the spec and walk it row by row.

### 2. Eager symmetry

The spec uses suffix conventions (a `S` suffix for Supplement events, an `R` suffix for Request events, etc.) for some events but not all. AI generation followed the pattern further than the spec actually does, producing codes like `MOOTS` (Option Trade Supplement) and `MORE`/`MORR`/`MORA` (Option Route Event/Rejected/Accepted) that are pattern-consistent but not real.

A second symmetry trap was the round number "50." Once a target of "all 50 CAT events" was set in documentation, the table was filled to 49–50 rows rather than to whatever count the spec actually defines. This is the same failure pattern as confabulation under exam conditions: when there's pressure to produce a complete answer and no penalty for getting individual items wrong, the answer gets completed by extension.

### 3. Self-signed audit artefacts

The repository included `audit/validate_report.md` and `audit/validate_entity_matrix.csv` claiming "All 10 VALIDATE gates (V01–V10) PASS" and "50 of 50 CAT events covered." Those files were generated by the same process that authored the reference table. They are not external verification; they are the author marking their own homework.

The wiki then cited those audit files as authoritative ("authoritative per-event ledger is `audit/validate_entity_matrix.csv`"), creating a confirmation cascade where each layer cited the next without anyone reaching the underlying spec.

### 4. Hedge-tag camouflage

Several places in the wiki and reference data carried `[VERIFIED]` or `[INFERRED]` tags. These signal "I checked this" but were applied without an underlying check. The presence of the tag made the claim look harder than it was, which is worse than no tag at all — readers (including the next AI run) treat it as already-verified.

### 5. Mixing two specs

CAT and CAIS are separate specifications with different document numbers, different reporting flows, and different data structures. CAT is event-based and CAIS is file-based (FDID Records and Customer Records, not "events"). The shipped table treated CAIS as another event family and invented codes (`CAIS_A`, `CAIS_C`, `CAIS_R`) to fill the slot. A glance at the CAIS spec table of contents would have caught this; the spec was not opened.

## Why the existing controls didn't catch any of this

| Control that existed | Why it failed |
|----------------------|---------------|
| Manual review by author | Single author. Same person who generated the table reviewed it. |
| `audit/validate_*` files | Generated by the same pipeline as the data they claim to validate. |
| Wiki "Verification status" sections | Hand-typed, never reconciled to source. |
| Cross-reference matrix in `docs/relationship_matrix.csv` | Built from the same fabricated reference data. |
| AI-authorship audit (`AUTHORSHIP_AUDIT.md`) | Looked for stylistic AI fingerprints (em-dashes, hedge tags, exhaustive parentheticals). Did not look at whether reference data was real. The earlier audit caught the *style* of fabrication but missed the *substance*. |

The pattern across all five: every check was authored by the same agent that authored the data being checked, and none of them required physical cross-reference to a primary source.

## The conditions that made it possible

The proximate cause was AI generation without a primary source. The systemic conditions were:

1. **A target count was named before the data was extracted.** "50 events" appeared in framing language (the README, the wiki Layout section) before the reference table was built, creating a slot to fill rather than a question to answer.

2. **Plausible scaffolding existed for fabrication.** CAT event codes share a regular shape (`M` + `E`/`O` + 2-letter action). A model can extend the pattern indefinitely; nothing in the format itself fails when a code is invented.

3. **Documentation reinforced rather than challenged the data.** The wiki's "All 50 CAT events are covered" did not say "we covered codes A, B, C, D... and we do not cover E, F, G." It treated coverage as a binary claim, which is exactly the kind of claim that's easy to bluff.

4. **No external diff existed.** A diff between the reference table and an authoritative source was never run because the authoritative source was never on disk.

## Controls that would have caught it

The controls in `guardrails/` are designed to catch this class of error before it ships:

| Control | What it catches | When it runs |
|---------|-----------------|--------------|
| `primary-sources/` directory with pinned PDFs | Construction without a primary source | Before any commit; commit fails if the PDF or its SHA-256 hash is missing |
| `validate_event_taxonomy.py` | CSV codes that don't appear in the PDF; PDF codes that don't appear in the CSV | Pre-commit hook + CI on every PR |
| `spec_pins.json` with `expected_event_count` | Drift in the total count | Same |
| `VERIFICATION_PROTOCOL.md` Rule 2 (extracted not authored) | Reference data being typed instead of extracted | Code review |
| `VERIFICATION_PROTOCOL.md` Rule 4 (counts come from validator) | Hand-typed coverage claims | Code review |
| `[VERIFIED]` / `[INFERRED]` discipline tied to citations | Hedge-tag camouflage | Code review |

If any of these controls had been in place, the v2.0.0 push would have failed. The validator alone would have surfaced 24 fabrications and 74 missing codes immediately.

## Lessons that generalize beyond this repo

1. **A target number is a vulnerability, not a feature.** Whenever documentation says "covers all N of N," somebody has to do the comparison. If the comparison isn't checked-in code, the claim is unfounded.

2. **AI-generated reference data needs an external diff. Always.** The diff is the only thing that constrains the output; absent the diff, the output drifts toward what looks correct rather than what is correct.

3. **Audits authored by the same agent as the data are not audits.** Independence is not optional. Either a different person, or a deterministic check against an external source, or both.

4. **Symmetry is suspicious.** Round counts (50 events, 16 reference tables, 10 VALIDATE gates), perfectly-balanced taxonomies, and exhaustive parallel structures should trigger extra scrutiny. Real specs are asymmetric.

5. **Hedge tags applied without a check are worse than no tag.** They signal verification that did not happen.

## What this audit does NOT cover

- Stylistic AI-authorship fingerprints (em-dashes, exhaustive parentheticals, etc.). Those are covered in `AUTHORSHIP_AUDIT.md`.
- The accuracy of the SQL DDL itself (column types, constraints, foreign keys). The DDL was not reconciled against the spec field definitions in this audit; it should be in a follow-on.
- The accuracy of the diagrams. Diagrams were generated from the same fabricated reference data and need to be regenerated against the corrected CSV.
- The DLT pipeline logic. The pipelines reference the (now-corrected) reference table, but quality gates may still encode fabricated assumptions.

A follow-on audit should reconcile the DDL field definitions against the spec field-specification tables (e.g. Table 64 for Option Order Route Event). That work is not in this remediation.
