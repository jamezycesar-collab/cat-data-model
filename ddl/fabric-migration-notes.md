## 2. Conversion script - `_delta_to_fabric.py`

Single-file Python 3 converter. Two independent transform functions:

### 2.1 `to_lakehouse(sql)` - minimal transform

Fabric Lakehouse accepts nearly all Databricks Delta DDL (USING DELTA, TBLPROPERTIES). The transform only removes what is unsupported or autonomous in Fabric:

| Remove | Why |
|--------|-----|
| `ZORDER BY (...)` | Fabric V-Order is automatic at write time |
| `CLUSTER BY (...)` (Liquid Clustering) | Not yet GA in Fabric Lakehouse as of the source SKILL cutoff |
| `delta.autoOptimize.*` TBLPROPERTIES | Fabric manages optimize/compact automatically |

Everything else (PARTITIONED BY, COMMENT on table + column, TBLPROPERTIES CDF flags, compression.codec, DECIMAL precision, BIGINT, DATE, TIMESTAMP) is retained verbatim.

### 2.2 `to_warehouse(sql)` - T-SQL re-emit

Full stateful parser that rewrites each `CREATE TABLE` body and swaps Databricks-only syntax for Fabric Warehouse T-SQL:

| Databricks Delta | Fabric Warehouse T-SQL |
|------------------|------------------------|
| `STRING` | `NVARCHAR(MAX)` |
| `TIMESTAMP` | `DATETIME2(7)` |
| `BOOLEAN` | `BIT` |
| `DATE'2024-01-15'` | `'2024-01-15'` (T-SQL implicit cast) |
| `DEFAULT current_timestamp` | `DEFAULT SYSUTCDATETIME` |
| `DEFAULT current_date` | `DEFAULT CAST(SYSUTCDATETIME AS DATE)` |
| `GENERATED ALWAYS AS (current_timestamp)` | `DEFAULT SYSUTCDATETIME` |
| `GENERATED ALWAYS AS (<expr>)` (non-timestamp) | *removed* - computed columns in Fabric Warehouse are limited; left as gap for review |
| column-level `COMMENT 'x'` | inline `/* x */` |
| table-level `COMMENT 'x'` | leading `-- <table>: x` |
| `USING DELTA` | *omitted* - implicit for Fabric |
| `PARTITIONED BY (...)` | *dropped* + re-expressed as `WITH (DISTRIBUTION = HASH(col))` when col is HASH-eligible, else `DISTRIBUTION = ROUND_ROBIN` with a `-- Consider HASH on...` hint |
| `TBLPROPERTIES (...)` | preserved as block comments after `);` (T-SQL noop - Fabric properties are configured via REST API / portal) |

### 2.3 HASH-eligibility filter

Fabric Warehouse rejects `DISTRIBUTION = HASH` on `NVARCHAR(MAX)` columns (including anything we converted from Delta `STRING`). The converter maintains an explicit allow-list of fixed-width DATE / DATETIME2 columns known to be good distribution keys:

```python
HASH_ELIGIBLE = {
 # DV2 bridge / business-vault partitions
 "trade_date", "report_date", "activity_date", "fact_date", "exposure_date",
 "contractual_settle_date", "actual_settle_date", "snapshot_date",
 "cat_reportable_date", "hour_ts", "effective_start", "effective_end",
 # Expanded-model DATE partitions
 "created_date", "effective_start_date", "event_date", "position_date",
 "submission_date", "expiration_date", "break_detected_date",
 "call_date", "correction_date", "sync_date",
}
```

Decision logic in `flush`:

1. Capture the full comma-separated list from `PARTITIONED BY (col1, col2,...)` into `partition_cols`.
2. Pick the first `partition_cols` entry that is in `HASH_ELIGIBLE` -> emit `DISTRIBUTION = HASH(col)`.
3. If no match, fall back to `DISTRIBUTION = ROUND_ROBIN` and append a human-readable comment:
 `-- Consider HASH on a fixed-width column; original partition: col1, col2`.

Enum-style partitions (`contract_type`, `asset_class`, `agency`, `venue_type`, etc.) are intentionally NOT in HASH_ELIGIBLE - they are NVARCHAR(MAX) and would be rejected by Fabric. ROUND_ROBIN is the correct default for these; re-sharding is a platform engineering platform-engineering decision.

## 3. Platform decision - Lakehouse vs Warehouse

Per skill guidance (section Fabric platform selection):

| Workload | Target | Rationale |
|----------|--------|-----------|
| Streaming ingest (CAT events, order lifecycle) | Lakehouse (Spark + Delta) | CDF, schema evolution, cheap append-only |
| Data Vault 2.0 Silver (Hubs/Links/Satellites) | Lakehouse | DV2 loads are append-heavy; Spark SQL compatible |
| PIT + Bridge rebuilds | Lakehouse | DLT append / MERGE friendly |
| Gold star schema for BI (FINRA CAT reports, Power BI) | Warehouse | Sub-second concurrency, T-SQL surface, DirectLake |
| Ad-hoc regulator extracts | Warehouse | T-SQL + OPENROWSET + stored procedures |

The two DDL targets are generated so either the Lakehouse or the Warehouse can host any given object when the workload shifts. For Day-1 deployment the recommendation is:

- **Silver = Lakehouse** (all DV2 objects)
- **Gold + Semantic Model = Warehouse** (facts, dims, aggregates for Direct Lake / Power BI)

Both read the same OneLake storage; cross-engine queries are via shortcuts.

Both engines read OneLake natively; the Warehouse reads Lakehouse tables via the automatic shortcut convention.

## 5. Known gaps & deferred items

The following Databricks-only features are **not** translated by the converter - they are either not supported in Fabric or require platform-level (not DDL) configuration:

| Deferred item | Where raised | Owner workstream |
|---------------|--------------|------------------|
| Change Data Feed (CDF) activation | TBLPROPERTIES comment in output | (DLT pipelines - Fabric CDC is configured per pipeline, not per table) |
| ZORDER / Liquid Clustering | Stripped in `to_lakehouse` | (Fabric V-Order handles automatically) |
| `GENERATED ALWAYS AS (<non-timestamp expr>)` | Removed with no replacement | review - re-express as view or computed column in Fabric Warehouse |
| Identity columns (`GENERATED BY DEFAULT AS IDENTITY`) | Not present in source Delta DDL, but Fabric Warehouse has `IDENTITY(1,1)` if ever needed | n/a |
| Row-level security / dynamic data masking | Not in source; add in Warehouse via `CREATE SECURITY POLICY` | (governance) |
| Delta log retention, min reader/writer versions | Preserved as comment - set via Fabric REST API | (platform ops) |
| Foreign key / PK constraints | Delta doesn't enforce them; Fabric Warehouse supports them as **non-enforced** (metadata only) - not emitted by the converter to keep parity with Lakehouse side | review (PKs matter for Power BI semantic model hints) |

## 6. Deployment checklist

1. Create Fabric workspace with Lakehouse + Warehouse in the same workspace so the automatic SQL endpoint / shortcut works.
2. Execute `expanded_ddl_fabric_lakehouse.sql` and `dv2_ddl_fabric_lakehouse.sql` against the Lakehouse via the SQL Analytics Endpoint notebook (Spark SQL cell).
3. Execute `expanded_ddl_fabric_warehouse.sql` and `dv2_ddl_fabric_warehouse.sql` against the Warehouse via SSMS / Fabric Warehouse query editor.
4. Configure CDF per table via Fabric REST API (not DDL-controllable).
5. Apply RLS / DDM via the governance workstream using `CREATE SECURITY POLICY`.
6. Wire DLT pipelines that MERGE bronze -> silver using `record_source` as source tag.
7. Build Gold star schema in the Warehouse so Power BI Direct Lake can consume it.
